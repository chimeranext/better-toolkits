## ВҝQuГ© es un good SPIKE para launchpad?

Un SPIKE Гәtil para productizar en chimeranext Launchpad tiene **4 caracterГӯsticas**:

1. **Concrete pattern** вҖ” no "William deberГӯa revisar UX" sino "founders en stage=Ideation con industry=fintech tienen 60% mГЎs engagement cuando el intake pregunta traction signals ANTES de competitive landscape"
2. **Dogfood evidence** вҖ” claim fundamentado en вүҘ1 sesiГіn de dog-food con artefactos generados (startup-profile.md, etc.)
3. **Productization hypothesis** вҖ” una idea concreta de cГіmo productizarlo en chimeranext (UI flow, data model change, algorithm tweak...)
4. **Acceptance criteria** вҖ” cГіmo William puede validar si la hipГіtesis funciona post-implementation

SPIKES que NO cumplen estas 4 son mejores como notes o user feedback, no como SPIKE formal.

### Paso 1 вҖ” Identificar el learning

**FTS-1**: "ВҝQuГ© aprendiste del dog-food que podrГӯa valer productizarse en chimeranext Launchpad? DescribГӯlo en 1-2 oraciones (el 'concrete pattern')."

## Principios clave

- **NO crear issues sin confirmaciГіn explГӯcita** вҖ” default es generar draft, filear solo si user dice "sГӯ"
- **William como default assignee** (ID Linear: `8f14370d-3602-49e3-81f2-eeb05b965687`; email: `william@chimeranext.io`)
- **Parent issue = legacy-ticket por default** вҖ” mantiene trazabilidad con el tracking SPIKE del plugin
- **Labels consistentes**: `spike`, `launchpad`, `methodology-prototype` вҖ” William filtra por estos
- **No escalate prematurely**: si el learning es dГ©bil, sugerir "keep dogfooding mГЎs antes de SPIKE"

## Anti-patterns

- SPIKEs vagos tipo "mejorar UX" вҖ” rechazar, pedir specific pattern
- SPIKEs sin acceptance criteria вҖ” no hay forma de validar post-implementation
- Crear issues bypaseando William вҶ’ viola ownership del pillar (ver feedback memory)
- Usar SPIKE format para feature requests вҖ” si es clear feature (no investigation), deberГӯa ser feature issue, no SPIKE
